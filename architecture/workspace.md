# Workspace

Workspace identity, the daemon-owned registry, the immutable execution context,
and the path policy that prevents tools from escaping project roots.

This module owns the lowest-level abstraction at which the daemon reasons about
"which project is the agent working on". It is UI-, server-, and plugin-free
and lives in `crates/codegg-core/src/workspace.rs`.

The typed identity foundation in `crates/codegg-core/src/identity.rs` keeps
logical project identity separate from this workspace locator. A
`ProjectBinding` can associate one `ProjectId`/`RepositoryId` pair with many
`WorkspaceId` values, and optional `WorktreeId`/`NodeId` values describe the
execution location without changing project identity.

## Why it exists

The single-daemon multi-project orchestration roadmap replaces the implicit
"current directory = project identity" model with an explicit, typed
workspace binding. Phase 2 makes every persisted session point at exactly
one `WorkspaceId` and propagates an immutable `ExecutionContext` through
every daemon-owned execution path.

## Key types

- `WorkspaceId` — typed `String` newtype identifying a registered workspace.
  Generated by the registry at registration time; never derived from path
  text so renames and host moves do not rename identifiers.
- `WorkspaceRecord` — persisted registry entry (`canonical_root`,
  `display_name`, `time_created`, `time_last_opened`, `time_archived`).
- `WorkspaceRegistry` — daemon-owned registry. Deduplicates canonical roots
  via `get_or_register`; rejects nonexistent paths, file paths, and
  symlink-escaping paths. Hydrates from a `WorkspaceStore` and serves
  subsequent reads from an in-memory `DashMap` index.
- `WorkspaceStore` — persistence trait. Two implementations:
  `SqliteWorkspaceStore` (the production path) and
  `InMemoryWorkspaceStore` (tests).
- `ExecutionContext` — immutable, `Arc`-wrapped context carrying
  `workspace_id`, `workspace_root`, `session_id`, `allowed_read_roots`,
  `allowed_write_roots`, and a `CancellationToken`. Every daemon-owned
  execution path receives one of these through `TurnRunInput.execution`.

## Path policy

`ExecutionContext` exposes three helpers that should be the only way tools
and subsystems materialize a path:

- `resolve_relative_cwd(requested)` — picks a working directory. `None` and
  empty strings resolve to the workspace root; relative paths join the
  workspace root and canonicalize; absolute paths must lie under an
  explicitly allowed root.
- `resolve_read_path(requested)` — verifies and canonicalizes a read
  target. Existing paths canonicalize directly; paths that don't yet
  exist canonicalize the nearest existing ancestor and append the missing
  suffix. Relative paths must resolve under the workspace root.
- `resolve_write_path(requested)` — same rules as read, plus the
  destination must lie under `allowed_write_roots`.

`PathPolicyError` distinguishes `OutsideWorkspace`, `OutsideAllowedRoots`,
`UnsupportedAlias`, `CanonicalizationFailed`, `EmptyPath`, and `ContainsNul`
so callers can render precise diagnostics.

## Session binding

The authoritative relation is:

```text
Session -> ProjectId + WorkspaceId -> canonical root
```

`SessionBinding` expresses the typed internal relation. `project_id` and
`directory` remain as compatibility projections in storage and DTOs; the
directory is a filesystem locator, not a project identity. New daemon code
reads `workspace_id`. `CoreDaemon::bind_runtime_for_session`
resolves a `session_id` to a `SessionRuntime` through `SessionStore` +
`WorkspaceRegistry`. `TurnSubmit`, `AgentSelect`, and `ModelSelect` reject
sessions whose `workspace_id` is `NULL` until rebound via
`CoreRequest::WorkspaceRegister`.

## Storage

Schema migration v22 (`crates/codegg-core/src/session/schema.rs:809`) adds:

```sql
CREATE TABLE workspace (
    id TEXT PRIMARY KEY,
    canonical_root TEXT NOT NULL UNIQUE,
    display_name TEXT NOT NULL,
    time_created INTEGER NOT NULL,
    time_last_opened INTEGER NOT NULL,
    time_archived INTEGER
);

ALTER TABLE session ADD COLUMN workspace_id TEXT;
CREATE INDEX idx_session_workspace_repair ON session(workspace_id);
```

Existing sessions are lazily resolved on next access; their `directory` is
canonicalized into a workspace record. Sessions whose `directory` does not
canonicalize to a real directory remain unbound and are rejected at turn
submission.

## Protocol

- DTO: `WorkspaceSnapshot { workspace_id, canonical_root, display_name, active_sessions }` in `crates/codegg-protocol/src/dto.rs`.
- Requests: `CoreRequest::WorkspaceRegister { root }`,
  `CoreRequest::WorkspaceList { include_archived }`,
  `CoreRequest::WorkspaceArchive { workspace_id }`,
  `CoreRequest::WorkspaceSnapshotRequest { workspace_id }`.
- Responses: `CoreResponse::WorkspaceList { workspaces }`,
  `CoreResponse::WorkspaceSnapshot { workspace }`.
- `SessionSnapshot` carries `workspace_id` (new) and `directory` (compat).
- `ServerCapabilities` advertises `workspace_registration` and
  `workspace_snapshots`.

## Propagation

- `TurnRunInput` (`src/agent/turn_runtime.rs:124`) carries
  `execution: Arc<ExecutionContext>`. The turn runtime uses
  `execution.workspace_root` for LSP context, Git context, project
  instructions, prompt loading, and tool registry construction.
- `ToolRegistryOptions` (`src/tool/mod.rs:252`) carries `workspace_root`.
  `build_session_tool_registry` (`src/tool/factory.rs`) anchors `RunStore`
  at `workspace_root/.codegg/runs/` and propagates `workspace_root` into
  tool options.
- Subagents receive the parent turn's `ExecutionContext` so that nested
  tool calls and `BashTool` invocations remain inside the same workspace.
- `BashTool` resolves cwd from the execution context first; relative
  `workdir` joins the workspace root, absolute paths must fall under an
  explicitly allowed root, and there is no process-cwd fallback for
  daemon-originated requests.

## Static guard

`scripts/check_daemon_cwd_usage.py` rejects new uses of
`std::env::current_dir()` and `std::env::set_current_dir(...)` inside
protected modules:

- `src/core/`
- `src/agent/turn_runtime.rs`
- `src/agent/worker.rs`
- `src/tool/` process/filesystem executors
- `src/test_runner/`
- `src/python_script/`
- Git execution services

Existing legacy uses in tool `default()` constructors are explicitly
allowlisted. New production-path uses fail CI.

`scripts/check_identity_path_usage.py` is an opt-in, narrow guard seam for
future project-storage work. It checks for explicit path-derived `ProjectId`
constructors while leaving the current compatibility projections untouched;
it is intentionally not enabled as a repository-wide CI rule in this
milestone.

```bash
cargo test --test workspace_isolation

python3 scripts/check_daemon_cwd_usage.py
```
