## 1. Objective

Allow Tool Programs to submit and await safe scheduler-owned build, test, lint, and format jobs through the Tool Broker, with inherited deadlines, resource admission, cancellation, progress, structured reports, artifact preservation, and RTK-compatible output projection.

The program runtime must never spawn these commands directly or implement its own polling loop.

## 2. Readiness boundary

Hard dependency: M006 closure. Foreground program execution, structured tool values, artifacts, and read-only palette must be stable before adding heavy child work.

## 3. Current implementation evidence

- Scheduler executors already support test and managed argv jobs for build/lint/format/shell/managed process.
- TestRunner owns supervised test semantics, status, RunStore, bounded reporting, timeout, and stall timeout.
- ManagedProcessService owns process provenance, cancellation, timeout, output truncation, and process-tree termination.
- Bash command routing classifies build/test/lint/format intents and can submit scheduler jobs.
- RTK discovery and invocation modes exist, and Bash records RTK eligibility metadata, but scheduled job results do not pass through one shared typed projection layer.
- The program runtime currently supports broker calls but not explicit child-job ownership or progress heartbeat.

## 4. Invariants that must not regress

- The scheduler remains the only admission authority for build/test/lint/format work.
- A program call creates at most one child job for one idempotent submission identity.
- Child resource requests and exclusivity keys cannot be weakened by program-supplied input.
- Parent deadlines and authority monotonically narrow into child jobs.
- Parent cancellation propagates to all child jobs and their process groups.
- Raw stdout/stderr/test evidence is preserved in RunStore/artifacts before projection or compression.
- Native typed projectors are preferred; RTK is a bounded projection fallback and never changes exit status, test status, failure class, or artifact contents.
- Program parallelism cannot multiply scheduler permits or bypass workspace/build-cache contention policy.
- General shell, arbitrary command strings, Git mutation, and dependency installation remain direct-only.

## 5. Scope

### In scope

- Program-callable contracts for bounded build, test, lint, and format operations.
- Child-job submission, correlation, wait, progress, cancellation, and terminal normalization.
- Structured build/test result schemas and artifact handles.
- Deadline/stall/resource inheritance.
- Bounded matrix execution through program loops/parallel groups.
- Shared output projection service with native projector first and RTK fallback.
- Cache/build-resource and retry semantics.

### Explicitly out of scope

- Arbitrary Bash/shell/terminal calls.
- Git writes, commits, pushes, worktree creation, or merge.
- Mutating Python transform programs.
- Background parent notification.
- External CI backends.
- Changing repository-wide test thread policy.

## 6. Required production changes

### Child-job broker adapter

Add broker routes that translate typed tool values into canonical `NewJob` submissions:

- `build`: validated argv/profile selected from existing command planner or explicit allowlisted project command;
- `test`: canonical `JobPayload::Test` with scope, cwd, timeout, stall timeout, and report bounds;
- `lint` and `format`: managed argv payloads with explicit semantics and resource profiles.

Do not accept a raw shell command string from the program. Inputs must be typed and either generated by trusted planners or validated against bounded argv policies.

Each call records:

- parent program/call/job/attempt IDs;
- child job and attempt IDs;
- submission fingerprint/key;
- inherited workspace/session/turn/agent context;
- effective resources/deadline/timeout/stall timeout;
- terminal run/artifact/projection references.

### Scheduler wait and progress

- Use scheduler event/wait APIs rather than repeated status-tool calls.
- Child progress refreshes program heartbeat only when actual child progress/heartbeat advances.
- Program cancellation requests child cancellation once and joins terminal reconciliation.
- Child terminal state maps to typed `ToolValue`; failed tests remain a successful tool invocation containing `status=failed`, while infrastructure failure is a broker/tool error. Preserve this distinction.

### Result schemas

Define stable schemas:

- Build/lint/format: status, exit code/signal, duration, command identity, diagnostics summary, stdout/stderr artifacts, truncation, changed files where applicable, run/job handles.
- Test: status, framework/runner, counts, failed tests, duration, stall/timeout/cancel state, concise failure evidence, full report/log artifacts, rerun metadata.

Do not infer success from string matching.

### Output projection service

Introduce one post-execution projection pipeline:

```text
raw RunStore artifacts
    -> native typed projector when available
    -> RTK generic command-output projector when eligible
    -> bounded fallback truncation/error retention
    -> model display plus artifact handles
```

Required RTK rules:

- RTK receives only approved bounded input and runs through managed process policy.
- RTK failure falls back to native/bounded output and cannot fail the underlying job.
- RTK output is untrusted projection, never authoritative status.
- Record projector implementation/version, input/output bytes, duration, truncation, and fallback reason.

### Retry and matrices

- Build/test calls default to safe-repeat only where the existing job contract supports it.
- Retry only transient infrastructure failures, not deterministic compile/test failures.
- A program may express a bounded matrix, but each child consumes normal scheduler resources and global fairness.
- Deduplicate identical child submission identities within a program and replay completed child results after restart.

## 8. Failure, cancellation, restart, and contention semantics

- Submission failure before child job creation is a broker failure and may retry only when transient.
- Enqueue ambiguity is resolved through submission key lookup; never submit a second child blindly.
- Compile/test failure is returned as a typed terminal value and does not trigger infrastructure retry.
- Cancellation while queued cancels the child job; while running it propagates through executor/process group and is joined.
- Program timeout cannot be extended by a child timeout. Child deadlines are no later than the program deadline.
- Stall detection uses child runner/process progress; a quiet but healthy process must follow existing supervisor semantics.
- Restart reloads child job identity from the call ledger and waits/reconciles it rather than resubmitting.
- Parallel matrices remain bounded by program width, scheduler admission, process slots, memory hints, and exclusivity keys.

## 9. Compatibility and migration

- Existing Bash/test direct tools remain available and use the same scheduler subsystems.
- Programmatic contracts are additive; raw shell compatibility is not inherited.
- Existing test/build projections may be wrapped into typed values while preserving current direct display.
- RTK remains optional and configuration-compatible; no installation becomes mandatory.
- If format is mutating under current implementation, keep it direct-only until an explicit safe check-only contract exists.

## 10. Required tests

### Focused unit tests

- input/output schemas and status distinction;
- child submission fingerprints and resource/deadline inheritance;
- RTK/native/fallback projection rules;
- transient versus deterministic retry classification.

### Integration tests

- program submits/awaits passing and failing build/test/lint jobs;
- exact program-call-child-job-attempt-run-artifact correlation;
- failure evidence remains expandable;
- no model-visible polling calls.

### Restart and recovery tests

- restart while child queued/running/completed-before-checkpoint;
- replay returns completed child result without duplicate process;
- ambiguous enqueue resolved by submission identity.

### Contention and cancellation tests

- bounded build/test matrix under global permits;
- cancellation of queued and running child groups;
- build-cache/workspace exclusivity and serial test policy;
- no process, task, permit, or artifact-writer leakage.

### Security and negative tests

- raw shell/argv injection rejection;
- attempted dependency install/network command;
- forged child job ID or cross-workspace result;
- RTK failure/malformed output cannot alter authoritative status;
- oversized logs/reports remain artifact-backed.

## 13. Acceptance criteria

1. Programs submit safe typed build/test/lint operations only through scheduler jobs.
2. Child jobs inherit authority, workspace, deadline, resources, cancellation, and correlation.
3. Restart/cancellation never duplicates a child process or strands it.
4. Test/compile failure is distinct from infrastructure failure and remains fully evidenced.
5. Raw output is preserved; native/RTK projection cannot alter status.
6. Bounded matrices respect global resource and contention policy.
7. No arbitrary shell or unsafe mutation becomes program-callable.
8. No unresolved high or medium finding remains.

## 14. Stop conditions

Stop and report if M006 is not closed, child job submission cannot be made idempotent, current test/build executors bypass cancellation/resource policy, RTK would become status-authoritative, or safe typed argv cannot be separated from arbitrary shell execution.
