# Phase 10 Plan: Frontend-Neutral Plugin UI Events

## Objective

Extend Codegg’s core/TUI protocol event flow so plugin-generated UI effects can move through daemon, socket, stdio, embedded TUI, and future GUI/web clients without becoming ratatui-specific.

The result should be a frontend-neutral path for `codegg_protocol::ui::UiEffect` payloads. TUI should render them through `PluginUiState` and `PluginUiRenderer`; other clients should be able to render, degrade, or ignore them based on advertised capabilities.

## Current State

Earlier phases introduced:

- `codegg_protocol::ui::{UiNode, UiEffect}`;
- TUI-side `PluginUiState` and renderer adapter;
- process command response application through TUI command dispatch;
- plugin runtime/service paths;
- core lifecycle hooks that may emit `PluginResponse` values containing UI effects.

However, the remote TUI protocol and core event stream are still primarily string/snapshot-oriented. Plugin UI effects need a typed transport path.

## Design Decision

Session-scoped plugin UI should travel as core/protocol events. Local-only UI feedback can remain TUI-local.

Use this rule:

- If the plugin effect is generated by a core lifecycle hook, installed plugin, or session-scoped command, emit it through core events or remote TUI messages.
- If the effect is generated by a purely local TUI command that has no core/session meaning, TUI may apply it locally.

The implementation should prefer core events for anything that should survive multi-frontend mode.

## Files to Modify

### `crates/codegg-protocol/src/tui.rs`

Add a typed remote message variant for plugin UI effects.

Recommended:

```rust
PluginUiEffect {
    session_id: Option<String>,
    plugin_id: String,
    invocation_id: Option<String>,
    effect: crate::ui::UiEffect,
}
```

If there is already a generic event envelope, embed this as a payload type rather than adding a one-off path. The key is that the payload must be typed and serde-compatible.

### `crates/codegg-protocol/src/core.rs`

If `CoreEvent` exists and is the better session-scoped event stream, add:

```rust
PluginUiEffect {
    session_id: Option<String>,
    plugin_id: String,
    invocation_id: Option<String>,
    effect: crate::ui::UiEffect,
}
```

or a more generic:

```rust
UiEffect(UiEffectEnvelope)
```

Do not duplicate shapes in incompatible ways. Prefer a shared `UiEffectEnvelope` type in `ui.rs` or `plugin.rs` if both core and TUI protocols need it.

### `crates/codegg-protocol/src/ui.rs`

Add a shared envelope if useful:

```rust
pub struct UiEffectEnvelope {
    pub session_id: Option<String>,
    pub source: UiEffectSource,
    pub invocation_id: Option<String>,
    pub effect: UiEffect,
}

pub enum UiEffectSource {
    Plugin { plugin_id: String },
    Core,
    Tui,
}
```

Keep source metadata small. Avoid embedding root crate session structs.

### `src/core/daemon.rs`

When plugin lifecycle hooks produce `PluginResponse.effects`, publish them through the core event/event log path if they are session-scoped. Ensure socket/daemon subscribers can receive them.

### `src/tui/app/mod.rs`

Update remote event handling to consume plugin UI effect messages and call `apply_plugin_ui_effect`.

Update snapshot building only if needed. Avoid stuffing structured plugin UI into the current `dialog: Option<String>` field. If snapshot resync should include plugin UI state, add typed fields in protocol; otherwise use event replay/resync follow-up.

### `src/tui/runtime/command_dispatch.rs`

Keep local `TuiCommand::PluginUiEffect` support. This remains useful for embedded/local effects. But if an effect arrives through remote/core protocol, it should land in the same `App::apply_plugin_ui_effect()` path.

## Client Capability Negotiation

Extend client capabilities to advertise plugin UI support.

Recommended fields or string capabilities:

- `ui.dialog`
- `ui.toast`
- `ui.panel`
- `ui.status_item`
- `ui.table`
- `ui.markdown`
- `ui.code`
- `ui.progress`

If current capability negotiation uses a vector of strings, add these strings. If it uses a struct, add optional booleans with serde defaults.

## Degradation Rules

Add a deterministic degradation helper in protocol or TUI/client utility code.

Recommended rules:

- unsupported dialog -> chat block or toast summary;
- unsupported panel -> chat block;
- unsupported table -> markdown table or key-value lines;
- unsupported status item -> omit unless important;
- unsupported markdown -> plain text;
- unsupported code -> plain text with language header;
- unsupported progress -> text percentage.

Core should not need to know each client’s rendering internals. It should either send full effects and let clients degrade, or degrade only when producing client-specific output after capability negotiation.

## Event Ordering and Replay

Plugin UI effects should preserve ordering relative to the plugin invocation that produced them.

Add metadata:

- invocation id;
- plugin id;
- optional sequence number if event log already has one;
- session id.

If the current event log already sequences events, rely on that. Do not invent a second ordering system unless necessary.

For reconnect/resync:

- toasts do not need replay unless recent event replay already handles them;
- open dialogs/panels/status items may need state reconstruction;
- consider including persistent plugin UI state in snapshots only for durable surfaces such as panels/status items;
- transient dialogs can be event-only initially.

## Security and Policy

Not every plugin should be allowed to open every UI surface.

Add or enforce capability checks:

- command/plugin declares output surfaces;
- lifecycle hook response effects are filtered against declared permissions/capabilities;
- plugin cannot spoof another plugin’s UI surface id unless namespaced;
- plugin UI ids should be prefixed/namespaced by plugin id at application time if not already.

For remote clients, do not let plugin UI bypass permission dialogs or security prompts. Permission/question/security dialogs keep higher modal priority.

## Tests

### Protocol tests

- `UiEffectEnvelope` serde round trip;
- remote TUI `PluginUiEffect` message serde round trip;
- core event plugin UI serde round trip;
- old clients can ignore unknown variants if protocol currently supports that; if not, document version bump behavior.

### TUI tests

- remote plugin UI effect applies to `PluginUiState`;
- plugin dialog effect does not displace permission dialog priority;
- unsupported node degrades gracefully;
- panel/status effects are stored;
- duplicate/un-namespaced plugin ids are normalized or rejected.

### Core/daemon tests

- lifecycle hook response with UI effects publishes a plugin UI event;
- disabled plugin produces no UI effect;
- effect order is preserved across multiple effects;
- client without plugin UI support can ignore or receive degraded output.

### Capability tests

- plugin declaring only chat output cannot open a panel;
- status widget effect requires status capability;
- panel effect requires panel capability;
- malformed/oversized UI payload is rejected or degraded.

## Acceptance Criteria

- Protocol has a typed UI effect envelope or equivalent message/event variant.
- Core/session-scoped plugin UI effects can be published to subscribers.
- TUI consumes remote/core plugin UI effects through `App::apply_plugin_ui_effect()`.
- Client capability negotiation includes plugin UI support flags or capability strings.
- Unsupported UI effects degrade deterministically or are safely ignored.
- Plugin UI effects are policy/capability filtered.
- Tests cover protocol serde, TUI application, core event emission, capability filtering, and degraded/unsupported clients.

## Non-Goals

- Do not build full GUI/web frontend rendering.
- Do not add forms, arbitrary drawing, or plugin key handlers yet.
- Do not implement plugin management UX; that is Phase 11.
- Do not make all first-party dialogs use `UiNode` yet.
- Do not require snapshots to fully reconstruct every transient toast/dialog unless event replay already supports that.

## Risks and Mitigations

### Risk: Plugin UI bypasses critical dialogs

Mitigation: permission/question/security dialogs keep higher priority. Plugin dialogs wait or are queued.

### Risk: Remote protocol becomes cluttered with one-off variants

Mitigation: use a generic `UiEffectEnvelope` if both core and TUI need the payload.

### Risk: Unsupported clients break on new events

Mitigation: use protocol versioning/capability negotiation and ignore/degrade behavior.

### Risk: Plugin spoofing UI ownership

Mitigation: namespace IDs at application time using source plugin id and reject cross-plugin updates.

## Handoff Notes for Phase 11

Once frontend-neutral UI effects are transportable, plugin management commands can use the same UI path to render `/plugins`, `/plugin-info`, install diagnostics, permission summaries, and trust/runtime labels.
